# Auth runbook — chunk 5 (Better Auth + tenant-aware email)

## Scope

Chunk 5 shipped:

- Better Auth (`better-auth@1.6.5`) with cookie sessions + the `bearer`
  plugin + `magic-link` plugin. Email+password sign-up with verification.
- Tenant-aware transactional email via `src/server/email/send-tenant-email.ts`.
  Every link is rendered against the resolved tenant's `primaryDomain` —
  never from a Host header or any caller input. The policy is enforced by
  the function signature: callers pass a `Tenant` object, not a raw URL.
- Host-only session cookies (no `Domain` attribute) because tenants live on
  different eTLDs.
- Breached-password v0 via a committed top-passwords list, wired through
  BA's `hooks.before`.
- Redis sliding-window rate limit primitive (`src/server/auth/rate-limit.ts`)
  — chunk 6 adapter wraps it at the BA hooks layer.
- Service-layer identity seam: `resolveRequestIdentity(headers, tenant)`
  returns `{ anonymous | session | bearer }`. BA types are hidden behind
  this seam; chunk 6's tRPC context imports only this + `resolveMembership`.

Membership resolution (`src/server/auth/membership.ts`) returns `null` for
users without a tenant membership row — that's the legitimate state for
customers. Admin roles (`owner | staff | support`) get a row per tenant.

## Cookie policy — why host-only

Tenants live on distinct eTLDs (`brand-a.com` vs `brand-b.com`) per prd.md
§3.1. Two consequences:

- A `Domain=brand-a.com` cookie does nothing on `brand-b.com` — best case.
- A `Domain=.example.com` cookie (if we ever shared a parent) would leak a
  session across tenants — worst case.

Host-only cookies (no `Domain` attribute) are the only safe default.
Configuration: `advanced.crossSubDomainCookies` is unset / `undefined` in
`src/server/auth/auth-server.ts`. A Playwright test
(`tests/e2e/auth/cookie-scope.spec.ts`) asserts the emitted cookie has no
leading dot and its `domain` matches the exact host.

`Secure` is enabled only when `BETTER_AUTH_URL` starts with `https://`.
Local http://localhost:5001 testing and `pnpm start` from a built bundle
do not need Secure; staging / production do (auto because BETTER_AUTH_URL
is https).

## Link-host derivation

`sendTenantEmail({ tenant, to, locale, template, params })` derives the
link host from `tenant.primaryDomain` and the sender From-address from
`tenant.senderEmail`. A function signature like this **is** the security
boundary: there is no `host` or `request` parameter, so a caller cannot
poison the link.

BA's `sendVerificationEmail` / `sendMagicLink` callbacks receive a `request`
from Better Auth. We pass that request's Host into `resolveTenant(host)`.
The resolver queries the DB via `app_tenant_lookup` (narrow column grant)
and returns a `Tenant` — or null for unknown hosts, which causes the
callback to throw before sending. Unknown hosts fail closed.

The locale that drives email subject and body is read from the request's
`Referer` path prefix (`/en/...` or `/ar/...`). `Accept-Language` is the
fallback. The auth form's fetch explicitly sends `accept-language: ${locale}`
but WebKit can append its own value, so we prefer the Referer path (the
form is always hosted under `/{locale}/...`) for determinism.

## Magic-link token hashing

- Plaintext: 32 bytes of base64url, generated by BA.
- Storage: base64url-encoded HMAC-SHA-256(`TOKEN_HASH_PEPPER`, token), wired
  via BA's `storeToken: { type: 'custom-hasher', hash: hashMagicLinkToken }`.
  A DB-only compromise yields hashes, not usable magic links.
- TTL: 10 minutes. `allowedAttempts` default = 1 (single-use).
- Same pepper as PATs (`TOKEN_HASH_PEPPER`). Same threat model (offline
  brute-force of high-entropy tokens), so sharing the pepper is consistent;
  a KMS rotation procedure would rotate both simultaneously.

Known gap: email-verification tokens are NOT hashed in BA v1.6.5 — they are
signed JWTs (HS256 with `ctx.context.secret`). BA hashes for email-verify
would require forking the plugin. The signed-JWT design is security-
equivalent for the threat model at hand: DB compromise still does not let
an attacker forge tokens (the signing key is env-only, not DB-stored).
Rotation of `BETTER_AUTH_SECRET` invalidates all outstanding verify links
— that is acceptable at our TTL (24h) and traffic volumes (Phase 0 tiny,
Phase 1 still small). Documented here rather than worked around.

## Password policy

- Minimum length: 10 characters (`emailAndPassword.minPasswordLength`).
- No complexity rules — length beats class-mix every study.
- Breached-password check via `src/server/auth/breached-passwords.ts`,
  a tiny in-memory Set seeded from
  `src/server/auth/data/top-common-passwords.json`. Runs in a BA
  `hooks.before` on `/sign-up/email`, `/change-password`, `/reset-password`.
  TODO(phase-6-hardening): migrate to a serialized bloom filter over
  HIBP's top-10k, OR enable the `haveIBeenPwned` BA plugin once we accept
  the runtime dependency on the HIBP range API.

## Rate limits

- BA's built-in rate limiter is **disabled** (`rateLimit: { enabled: false }`).
  We own rate limiting at our own adapter layer using
  `src/server/auth/rate-limit.ts` (Redis ZSET sliding window).
- Default policy for `/api/auth/*` (to be wired by chunk 6): 5 attempts /
  15 minutes per (IP, email) tuple.
- Threshold override for tests: pass a custom `limit` / `windowSeconds`
  into `checkRateLimit` — or in production, a per-tenant config row.

## Dev seed + Playwright setup

- `pnpm db:seed:dev` / `scripts/seed-dev-tenant.ts` upserts a tenant row
  for `localhost:5001` with `defaultLocale=ar`, `senderEmail=no-reply@localhost`.
  Idempotent via `ON CONFLICT (primary_domain) DO UPDATE`.
- `tests/e2e/global-setup.ts` runs this seed + clears Mailpit before the
  whole test run.
- The DB-backed resolver consumes this tenant row; the `ALLOW_TENANT_FALLBACK`
  dev-fallback is retained but NO LONGER required for tests.

## Playwright web-server choice

E2E tests run against the **production build** (`pnpm build && pnpm start`),
not `pnpm dev`. Next.js dev-mode compiles routes on demand and occasionally
triggers HMR page-reloads that race the test's own navigation — this was
observed on the WebKit/iPhone-14 projects where a hot-update cancelled a
just-initiated `window.location.assign` to `/verify-pending`. The built
server is deterministic. Set `PLAYWRIGHT_USE_DEV=1` to override back to
`pnpm dev` for rapid iteration.

## Auth forms are real `<form>` elements — password-manager invariant

Signup (`src/app/[locale]/signup/signup-form.tsx`) and signin
(`src/app/[locale]/signin/signin-form.tsx`) are real `<form>` elements
with `button[type="submit"]` and a single `onSubmit` handler on the form.
Do **not** rewrite them as `<div>` + `button[type="button"]` + `onClick`
— password managers (1Password, Chrome's built-in manager, Safari
Keychain, Firefox) only offer to save credentials when they observe a
real `<form>` dispatching a real `submit` event that wraps recognisable
`autocomplete=email` + `autocomplete=new-password` (signup) or
`autocomplete=current-password` (signin) inputs. A store where save
prompts silently fail to appear is a conversion-killer.

The magic-link button on the signin page is deliberately OUTSIDE the
password form. It hits a different endpoint and placing it inside
would both (a) submit the password form on click and (b) confuse
password managers into suppressing the save prompt because two
submit-like controls compete for the same form.

### Why `onSubmit`-only (no `onClick` on the button)

A `button[type=submit]` inside a form fires **both** the button's
`onClick` AND the form's `submit` event on a single click. During the
chunk-5 build, a version with both handlers double-fired the sign-up
fetch on mobile WebKit. The correct fix is exactly one handler path:
`onSubmit` on the form, no `onClick` on the submit button. The button
itself carries no handler; clicking it, pressing Enter in a field, or
calling `form.requestSubmit()` all converge on the form's single
`onSubmit`. `e.preventDefault()` runs BEFORE any `await`, so the
browser does not GET-navigate while the fetch is in flight. Password
managers observe the submit event before `preventDefault` takes
effect — prevented submits still trigger save prompts.

### Invariant guard

`tests/e2e/auth/form-semantics.spec.ts` asserts, on both auth pages,
in both locales, on every Playwright project:

- Exactly one `<form>` on the signup / signin page.
- Email + password inputs with the correct `autocomplete` tokens live
  inside that form.
- The submit button is `form button[type=submit]`.
- On `form.requestSubmit()` a real `submit` event fires and is
  `defaultPrevented === true` (i.e., the handler intercepted it).
- Pressing Enter in the email input dispatches the native submit event.
- On signin, the magic-link button is NOT a descendant of the password
  form.

This is the closest machine-checkable proxy for "a password manager
will recognise this form". Keep it passing.

## Membership on signup — customers get none

`resolveMembership(userId, tenantId)` returns `null` for customers. Only
owner / staff / support users get a `memberships` row. Callers that require
admin-scoped authorization MUST check the return value and treat `null`
as "deny". This is not a bug; it's the prd.md §3.6 model: users belong to
the platform, not a tenant, so a single shopper can have carts and orders
across multiple tenants without any membership row ever being created.

## Rate limit policy (chunk 6 block 5)

`src/server/auth/rate-limit-auth-hook.ts` wires the Redis sliding-window
primitive into BA's `hooks.before`. Every auth endpoint in `AUTH_LIMITS`
passes through a two-tier check:

- **Per-IP** bucket key `auth:{tenantId}:{path}:ip:{ip}`. IP read
  from `x-real-ip` ONLY (the reverse proxy sets this); falls back to
  `unknown-ip` when the header is absent. `x-forwarded-for` is
  DELIBERATELY ignored — see "Traefik / proxy requirement" below.
- **Per-identity** (email) bucket key `auth:{tenantId}:{path}:email:{lc-email}`.
  Only applied on paths whose policy sets `identityKey: 'email'`.

Per-tenant prefix is the **isolation boundary** — one tenant's attack
cannot lock another tenant's users.

Current limits (see `AUTH_LIMITS` for the source of truth):

| Path                 | Per-IP        | Per-email     | Notes                                 |
|----------------------|---------------|---------------|---------------------------------------|
| /sign-up/email       | 20/min        | 5/15min       | Catches enumeration                   |
| /sign-in/email       | 20/min        | 5/min         | Catches credential stuffing + spray   |
| /forget-password     | 10/5min       | 3/hour        | Expensive email send; tight per-email |
| /reset-password      | 20/min        | — (IP-only)   | Token-gated; user may retry           |
| /sign-in/magic-link  | 10/min        | 3/15min       | Send-side, expensive                  |
| /magic-link/verify   | 30/min        | — (IP-only)   | Token-gated verify; loose             |

**Reject outcome:** `APIError('TOO_MANY_REQUESTS', { code: 'RATE_LIMITED' })`.
Every rejection writes an `auth.rate-limit-exceeded` audit row
(`error = '{"code":"rate_limited"}'` per the block-2 closed-set
invariant). The audit payload carries structural facts only
(`path`, `ipLimited`, `emailLimited`) — never the raw request body,
which may include a Tier-B email.

**Fail-closed on Redis outage:** `checkRateLimit` throws when the
pipeline returns null; the hook converts that to
`APIError('SERVICE_UNAVAILABLE', { code: 'RATE_LIMITER_UNAVAILABLE' })`.
The auth request does NOT pass through. If you see 503s with that code
in production, the primary lever is Redis health; secondarily, check
the `ratelimit:auth:*` key count in Redis for a runaway bucket.

**Traefik / proxy requirement (High-02):** The reverse proxy (Traefik
under Coolify) MUST set `X-Real-IP` to the client's true IP. Traefik
sets this by default when routing; confirm via `curl -v` against the
deployed URL that the header arrives at the app. If the header is
absent, every per-IP rate-limit bucket resolves to a single global
`'unknown-ip'` bucket — auth requests from ANY client will compete
for that budget, which is the correct fail-closed behavior but will be
noisy for real users.

We deliberately do NOT read `X-Forwarded-For`: Traefik's default is
append-mode, and the first entry of a client-submitted XFF header is
attacker-controlled. Reading `X-Real-IP` (which the proxy writes and
re-sets, ignoring any client value) is the only safe option. A unit
test at `tests/unit/auth/rate-limit-wire.test.ts` adversarially asserts
that XFF alone yields `'unknown-ip'` — the attacker-cannot-inject-IP
invariant.

**Per-email bucket normalization (Medium-02):** The per-email
rate-limit bucket key runs the caller-supplied email through
`normalizeEmailForBucket()` — lower-case + NFKC + strip plus-alias
suffix of the local-part. `alice@gmail.com`, `alice+1@gmail.com`, and
`alice+99@gmail.com` all share one bucket. Plus-alias rotation does
NOT defeat the per-email tier. Visual homoglyphs (e.g. Latin 'i' vs
Turkish dotless 'ı') stay in distinct buckets — those are semantically
distinct inputs. The normalized form is ONLY used for the bucket key;
Better Auth receives the raw caller-supplied email unmodified.

**Dev-bypass for `unknown-ip`:** `pnpm dev` has no reverse proxy in
front, so every caller resolves to `unknown-ip` (the `extractIp`
fallback). Under a strict per-IP cap this collapses dev usability —
any developer clicking around signup/signin in a browser hits the cap
in a few attempts. The hook therefore skips the per-IP tier when
`ip === 'unknown-ip'` AND `NODE_ENV !== 'production'`. The
per-identity (email) tier STILL fires under those conditions, so
credential-stuffing dev tests remain meaningful. In production the
Traefik/Coolify proxy MUST set `x-real-ip` — if it doesn't, we
fail closed (the bypass is gated on NODE_ENV, not on a missing proxy
in prod).

**Playwright global-setup flushes rate-limit buckets.**
`tests/e2e/global-setup.ts` does a `SCAN ... MATCH ratelimit:auth:* DEL`
at suite start to give every run a clean slate.

**E2E bypass — prod-unreachable by design.**
Playwright runs the production *build* (`pnpm build && pnpm start`)
which sets `NODE_ENV=production`, so a NODE_ENV-only gate would block
the bypass under E2E. Instead the bypass is DOUBLE-GATED on two
independent env vars:

- `APP_ENV === "e2e"` — a separate "deployment target" dimension owned
  by this repo (NOT by Next.js).
- `E2E_AUTH_RATE_LIMIT_DISABLED === "1"` — an explicit opt-in flag.

Both must be present. Real prod deploys (Coolify) set neither, so the
bypass is unreachable there. Set by `playwright.config.ts` webServer
env and nowhere else — if someone copies `E2E_...` into a prod env file
the bypass still does not activate, because `APP_ENV !== "e2e"`.

E2E tests that need to exercise the limiter-engaged path (e.g. the
rate-limit-exceeded PII canary) set `x-dev-only-enforce-rate-limit: 1`
per request; the header is a no-op outside the double-gated E2E
bypass (APP_ENV=e2e && E2E_AUTH_RATE_LIMIT_DISABLED=1), so a prod
container never reads it.

**Critical — E2E bypass must NOT reach prod.** Any prod deploy whose
environment contains `APP_ENV=e2e` OR `E2E_AUTH_RATE_LIMIT_DISABLED=1`
is a misconfigured deploy. Coolify's production env config MUST set
neither. Chunk 9's CI workflow will lint Coolify env files for these
keys and fail the deploy if found. Chunk 9 also adds a hard-refuse
boot check: the server refuses to start in prod (NODE_ENV=production)
with either flag set. This note lives here so the boot check is not
forgotten during chunk-9 CI work.

## Audit wiring for auth events (chunk 6 block 7)

BA's `databaseHooks` in `src/server/auth/auth-server.ts` invokes five
audit call sites in `src/server/auth/audit-hooks.ts`:

- `userCreateAfter` → `auth.signup` (success).
- `userUpdateAfter` → `auth.verify-email` when `emailVerified` flipped true.
- `verificationCreateAfter` → `auth.magic-link.request` on
  `ctx.path === "/sign-in/magic-link"` only (password-reset verification
  rows land later).
- `sessionCreateAfter` → `auth.session.create` always; additionally
  `auth.magic-link.consume` on `ctx.path === "/magic-link/verify"`,
  sharing the same `correlationId` as the session.create row so
  operators can join the two.
- `sessionDeleteAfter` → `auth.session.revoke` with reason-enum
  `'user_signout' | 'system' | 'unknown'` derived from `ctx.path`.

The failure-path signup audit lives in `auth-server.ts`'s
`hooks.after` (endpoint middleware). It reads `ctx.context.returned`
for APIError-shaped throws and writes `auth.signup` with
`outcome: "failure"` + an `errorCode` mapped from BA's error message
(`USER_ALREADY_EXISTS` → conflict, 429 → rate_limited, etc.). No
`input`, no `after` — chain holds structure; Sentry holds detail.

**Tenant resolution at hook sites (Option B):** when the BA ctx is
null or `resolveTenant` returns null, SKIP the audit write and fire
a Sentry `audit_write_failure` alert with
`reason: "tenant_resolution_lost_at_hook"`. `APP_ENV=seed` suppresses
the Sentry send so dev seed scripts don't flood the alert channel.

**APP_ENV=seed sentinel:** dev seed scripts set `APP_ENV=seed` to
suppress the `audit_write_failure` Sentry send on nullable-ctx BA
hook fires. The Option-B audit-skip invariant is preserved; only the
Sentry alert is suppressed. Chunk-9 CI env-lint rejects `APP_ENV=seed`
in prod Coolify, same class as `APP_ENV=e2e`.

**Session-revoke reason-enum:** `'user_signout'` when
`ctx.path === '/sign-out'`, `'system'` when `ctx === null` (internal
code paths), `'unknown'` fallback. `'admin_revoke'` branch to be added
when the `/admin/revoke-session` endpoint lands (not block 7).

**Four-row shape on magic-link-first-consume:** `/magic-link/verify`
for a previously-unknown email triggers four sequential audit writes
— `auth.signup`, `auth.verify-email`, `auth.magic-link.consume`,
`auth.session.create` — each grabbing the per-tenant
`pg_advisory_xact_lock('audit_log:' || tenant_id)`. Under contention
this 4x-multiplies the lock window for one user-facing event. Not a
correctness issue (chain writes are fast; lock is per-tenant not
global) but worth knowing if operator sees magic-link-verify latency
spikes for a specific tenant.

**Correlation topology across the four magic-link rows:**
`auth.magic-link.consume` and `auth.session.create` share a
`correlation_id`; `auth.signup` and `auth.verify-email` are joined by
`actor_id + timestamp` (they don't share correlation_id because signup
precedes session creation in a different transaction boundary).
Querying by correlation_id reaches the consume+session pair; joining
via actor_id reaches all four. Phase 4 MCP-query authors hitting the
audit log via `search_audit_log` should pick the join strategy
accordingly.
